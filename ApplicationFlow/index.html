Building blocks

Game Modes

The game is a state machine, consisting of a defined set of modes as states, as well as transitions between them. This is implemented in the application flow package. Modes are per se abstract entities: They are essentially collections of Modules and their behavior is defined by the modules they contain.

Modules

Modules are stand-alone parts of the game. They should work by themselves, i. e. they should not require other modules to be present in a Unity project in order to run (except for the game master.)

NOTE: It is important not to circumvent this concept by creating ‘linking packages’ that closely couple modules to each other. Packages may be included in multiple Modules, but a shared package dependency should ideally not lead to tight coupling between these modules. Each module should instead contain separate instances of whatever exists in those packages. The core functionality of a module should be contained in one or more packages that are only used by that respective module. While these will likely link various packages, it should not lead to tight coupling between them, which would make them unusable for other packages!

Modules need to reside in a single scene. When the game transitions to another mode, scenes of needed but not yet loaded modules will be loaded and vice versa. Modules will need to communicate with each other at times. However, they should also not require each other to be in the same project or, in fact, exist. This is achieved through messaging.
